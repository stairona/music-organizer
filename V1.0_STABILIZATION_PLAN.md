## Planned Enhancements

### 1. Metadata Enrichment

**Goal**: Improve genre classification accuracy by leveraging additional audio metadata beyond the basic `genre` tag.

**Current Limitation**: `tags.read_genre()` only extracts the primary genre field. Many audio files contain useful complementary tags (BPM, year, artist, composer, mood, album) that could inform genre inference when genre is missing or ambiguous.

**Proposed Approach**:

1. **Extended Metadata Extraction** (`src/music_organizer/metadata.py`):
   - Use `mutagen` to extract:
     - `BPM` (beats per minute) → can suggest Electronic genres (e.g., 120-130 = House, 140-150 = Techno)
     - `date`/`year` → older years may indicate classical or legacy recordings
     - `artist`/`composer` → known artists can map to specific genres via lookup table
     - `album` → sometimes contains genre keywords
     - `comment` → freeform notes that might describe style
     - `mood` or custom frames in Vorbis comments
   - Return a `Metadata` dataclass with all extracted fields.

2. **Genre Inference Boost**:
   - In `classify.classify_file`, after reading genre, if genre is missing or low confidence:
     - Use BPM ranges to boost specific genre probabilities:
       - 100-120: House, R&B
       - 120-130: Deep House, Techno, Disco
       - 130-140: Trance, Progressive
       - 140+: Drum & Bass, Hardstyle
     - Use artist lookup table (configurable JSON) mapping artists to likely genres.
   - Combine signals: if path inference suggests "House" and BPM is 122, confidence increases.

3. **Config Enrichment**:
   - Allow user-defined "genre hints" in `~/.config/music-organizer/genre_hints.json`:
     ```json
     {
       "artist_genre_map": {
         "Calvin Harris": "Pop/Dance",
         "Daft Punk": "Electronic/House"
       },
       "bpm_ranges": {
         "house": [115, 130],
         "techno": [125, 140],
         "drum_and_bass": [165, 185]
       }
     }
     ```

**Impact**: Reduce "Unknown" classification rate, especially for files with missing or vague genre tags.

### 2. Enhanced Duplicate Detection

**Goal**: Beyond exact file hash matching, detect duplicates that are not bit-identical but represent the same musical work (e.g., different bitrates, vinyl rips, remasters).

**Current Limitation**: `file_sha256` comparison only catches exact duplicates. Two MP3s of the same song at different bitrates (128kbps vs 320kbps) will be treated as distinct and both copied/moved, wasting space.

**Proposed Approach**:

**Option A: Audio Fingerprinting (High accuracy, medium complexity)**
- Integrate a lightweight audio fingerprinting library like `chromaprint` (via `pyacoustid` or `fpcalc` subprocess).
- Compute fingerprint for each audio file (requires ~10-20 seconds per file but can be cached).
- Match fingerprints to a database; similar fingerprints indicate same recording.
- Requires additional dependency: optional feature.

**Option B: Perceptual Hashing (Simpler)**
- Use robust audio hashing based on spectral features (e.g., `librosa` to compute mel-spectrogram hash).
- Too heavy for this CLI tool; would add large dependency.

**Option C: Metadata-Based Duplicate Detection (Lightweight)**
- Consider files duplicates if:
  - Same artist and track title (from metadata)
  - Similar duration (±5 seconds)
  - Optional: similar file size within 20%
- This is fast and requires no heavy processing.
- Risk: false positives for different recordings of same song (covers, live versions). Could add "source" tag check.

**Recommended Implementation (C)**:
1. Add new collision policy: `"smart"` (in addition to existing "hash", "rename", "skip").
2. In `fileops.resolve_collision`, if policy is "smart":
   - Check SHA256 first → skip if identical.
   - If different, extract metadata (artist, title, duration) using `mutagen`.
   - Compare against destination file's metadata (if available). If artist/title match (case-insensitive) and duration within threshold, treat as duplicate (skip or rename based on sub-policy).
   - Optionally store metadata cache to avoid repeated extraction.
3. Command-line flag: `--on-collision smart` or `--on-collision smart:rename` etc.

**Impact**: Reduce wasted storage space from near-duplicate audio files, improve user satisfaction.

### 3. Additional Minor Improvements

- **Progress spinner**: For large libraries, show a spinner or ETA rather than just "Processed N/..."
- **Parallel processing**: Option `--jobs N` to process files in parallel (careful with I/O). Could speed up on SSDs.
- **Symlink support**: Option to create symlinks instead of copy/move for space-saving.
- **Better unknown diagnostics**: Produce a separate "unknown report" with top unknown filename tokens and suggestions for `PATH_KEYWORDS` additions.
- **Config validation**: Add a `music-organizer check-config` command to validate custom genre mappings.

## Backward Compatibility

- All new features will be opt-in via new CLI flags or config options.
- Existing `--on-collision hash|rename|skip` behavior unchanged.
- Metadata extraction will not fail if tags are missing; returns empty `Metadata`.
- Smart duplicate detection will fall back to hash if metadata extraction fails.
